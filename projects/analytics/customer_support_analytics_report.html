Завдання 1.1¶

В чому різниця запитів? Чим будуть відрізнятись отримані результати?

SELECT *
FROM table1
LEFT JOIN table2 USING (name)
WHERE table2.name = 'John'

SELECT *
FROM table1
JOIN table2 USING (name)
WHERE table2.name = 'John'

Відповідь:¶

В першому випадку для обєднання таблиць використали LEFT JOIN який теоретично виводить всі результати з table1 навіть якщо немає відповідника у table2, але при цьому використали фільтр для імені з table2 "WHERE table2.name = 'John'" , що відкидає з неї нульові результати. В другому випадку використано JOIN, що виводить всі записи, де name співпадає в обох таблицях також з фільтром по імені з table2 "WHERE table2.name = 'John'" Отже результат обох таблиць буде однаковим тому що фільтр table2.name = 'John' у WHERE блокує будь-які рядки без відповідника в table2

Аналіз Total Spend (ТС) по користувачах¶

Витягнути всіх користувачів, ТС яких вище за середній ТС у їхньому регіоні

SQL-запит (BigQuery )¶

WITH avg_by_region AS (
  SELECT 
    id_region,
    AVG(amount) AS avg_amount
  FROM `task.total_spend`
  GROUP BY id_region
) 

SELECT 
  t.id_user, 
  t.id_region, 
  t.amount
FROM `task.total_spend` t
JOIN avg_by_region a 
  ON t.id_region = a.id_region 
WHERE t.amount > a.avg_amount
ORDER BY t.id_region

Завдання 2¶

Проаналізувати, як можна покращити рівень обслуговування клієнтів.

Вхідні дані:¶

  • Кожен продукт має свою команду підтримки.
  • Кожне завдання старанний агент служби підтримки може виконувати до 5 хвилин з момента взяття в роботу.
  • З боку менеджменту є очікування, що в рамках кожного дня середній час від виникнення нового запиту користувача до відповіді має бути 15 хвилин. Також погано, якщо користувачі сервісу чекають на відповідь довше 45 хвилин.

Опис даних:¶

  • Таблиця events - дані про кожний запит до модерації. Містять в собі інформацію про сам запит та модератора, який його виконував.
  • id_request - унікальний ідентифікатор запиту;
  • moderator - унікальний ідентифікатор агента підтримки;
  • team - параметр команди (retail / wholesale);
  • request_time - час отримання запиту;
  • start_time - час початку обробки кейсу;
  • finish_time - час кінця обробки кейсу.

  • Аналіз середнього часу відповіді по годинах і динаміки кількості запитів.

Кількість невирішених задач щодня¶

Графік показує різницю між кількістю запитів, що надійшли, та кількістю вирішених протягом кожного дня.
У період жовтня–листопада динаміка була стабільною: кількість нових задач приблизно відповідала кількості вирішених.

Починаючи з кінця листопада деякі дні показують різкий приріст запитів (+400+), інші — масове закриття задач (до -500).
Це вказує на нерівномірне навантаження, можливі дії з масової обробки backlog або залежність від нерівномірного графіку роботи модераторів.

Накопичення невирішених задач¶

З жовтня по середину листопада backlog залишався стабільним і не перевищував 400 задач.
Однак наприкінці листопада почалося стрімке зростання невирішених задач —
досягнувши піку понад 2200 задач у середині грудня.

Це свідчить про системну затримку з обробкою запитів, ймовірно через:

  • перевищення кількості нових задач над можливістю їх обробки,
  • нерівномірну активність команди або зменшення залученості модераторів. Рекомендовано:
  • запровадити окрему команду або зміну для методичного очищення backlog,
  • моніторити рівень накопичення на регулярній основі як ключову метрику

    active_days=("request_date", pd.Series.nunique)
).reset_index()

def categorize_moderator(row):
    if row["total_requests"] > 100 and row["active_days"] > 5:
        return "Активні"
    elif row["total_requests"] > 20 or row["active_days"] >= 2:
        return "Середньо активні"
    else:
        return "Неактивні"

moderator_stats["категорія"] = moderator_stats.apply(categorize_moderator, axis=1)
labels = ["Активні", "Середньо активні", "Неактивні"]
sizes = moderator_stats["категорія"].value_counts().reindex(labels, fill_value=0)
colors = ["mediumseagreen", "khaki", "lightcoral"]

plt.figure(figsize=(8, 5))
plt.pie(sizes, labels=labels, autopct='%1.1f%%', startangle=90,
        colors=colors, wedgeprops={"edgecolor": "white"})
plt.title("Категорії модераторів (загально)")

plt.tight_layout()

Активність модераторів¶

Переважна більшість модераторів (77.6%) є активними — вони мають понад 100 оброблених запитів та щонайменше 5 днів активності. Однак майже 22.4% модераторів потрапляють до категорій середньо активних або неактивних:

  • 10.2% — середньо активні (21–100 задач або 2–5 днів активності),
  • 12.2% — неактивні (до 20 задач або максимум 2 дні активності).

Це означає, що близько п’ятої частини наявної модераторської сили не використовується ефективно.
Рекомендовано:

  • активізувати менш залучених агентів або переглянути доцільність їх участі,
  • оцінити можливість перерозподілу запитів між активними та менш активними модераторами.

Відповіді на запитання та Рекомендації¶

1. Наскільки служба підтримки відповідає очікуванням менеджменту? Як вона працює зараз?¶

Служба критично не відповідає очікуваному часу реагування (15–45 хв), однак повністю відповідає вимогам щодо часу обробки задач після старту.

  • Лише 13.6% задач вирішуються в межах очікуваного часу відповіді.
  • Лише 8.1% задач отримують відповідь швидше ніж за 15 хв — це може свідчити про відсутність чітких правил пріоритизації.
  • Понад 78% задач обробляються із затримкою більше 45 хв, що значно перевищує допустимі межі.

2. На яких агентів варто звернути увагу задля підвищення якості обслуговування?¶

Більшість агентів демонструють високу швидкість обробки, проте:

  • деякі агенти мають надто низьку кількість опрацьованих запитів;
  • важливо проаналізувати, чому частина агентів системно обирає лише "свіжі" задачі, ігноруючи старі.

Рекомендація: ввести рейтингову систему KPI, що включає:

  • середній час відповіді;
  • середній час обробки;
  • загальна кількість виконаних задач.

3. Чи варто збільшити команду підтримки? Якщо так — наскільки?¶

Проблема не стільки у нестачі людей, скільки в нерівномірному розподілі навантаження:

  • майже 80% задач порушують часові межі реагування;
  • є агенти, які за весь досліджуваний період закрили лише 1 задачу. Рекомендація: не лише розширити команду, а й:
  • перерозподілити навантаження;
  • створити спеціалізовану групу для опрацювання backlog.

4. Як змінити графік роботи служби підтримки?¶

Так, графік потребує оптимізації:

  • варто підсилити ранкові та нічні години;
  • запровадити спеціальний щоденний блок для очищення черги із застарілих задач.

5. Чи є вигода від об'єднання команд?¶

Об’єднання команд retail та wholesale не рекомендується, оскільки:

  • в них різні темпи надходження задач;
  • існує ризик втрати фокусу та пріоритетності.

Краще зберегти поділ, але створити:

  • крос-командну підтримку для обробки backlog у вільні години.

Ключові рекомендації:¶

  • Впровадити принцип черги FIFO для вирішення задач за часом надходження.
  • Реорганізувати команду на два напрями:
    • Оперативна група — забезпечує швидке реагування на нові запити;
    • Backlog-група — методично обробляє старі задачі за принципом FIFO.
